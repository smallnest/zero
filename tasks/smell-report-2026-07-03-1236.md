# Architecture Smell Report

**Project:** zero (`internal/agent/`)
**Scope:** `internal/agent/` package — 13 source files (6,288 LOC) + 34 test files
**Date:** 2026-07-03
**Analyzer:** smell skill

---

## Executive Summary

`internal/agent/` is the orchestration core of Zero: it owns the agent run loop, tool
dispatch, the sandbox permission state machine, hook dispatch, compaction, and prompt
assembly. The package is **well-tested** (34 test files against 13 source files — a
2.6:1 ratio), builds cleanly, passes `go vet`, carries **zero** `TODO/FIXME/HACK`
markers, and has disciplined naming (no `Manager`/`Helper`/`Util` grab-bags). Those are
real signs of a healthy, actively-maintained codebase.

The dominant problem is **structural concentration**, not rot. A single file,
`loop.go`, holds 2,736 lines and 84 functions spanning at least five distinct
responsibilities. Its entry point `Run()` is a 545-line method nested up to 7 tabs
deep, and `executeToolCall()` adds another 328 lines. The permission/sandbox subsystem
(~50 functions) lives intermixed with the core loop rather than in its own file or
package. Compounding this, the `Options` struct is a 41-field, 108-line parameter
object that is passed by value nearly everywhere, and the pair
`(permissionMode, options)` is threaded through 16 signatures even though
`options.PermissionMode` already carries the same value.

None of this breaks today, but it raises the cost of every future change to the loop
and makes the permission logic hard to test and reason about in isolation. The top
fixes are all **mechanical extractions** — low risk, high readability payoff — because
the behavior is already pinned down by the strong test suite.

**Top 5 issues:**
1. 🔴 God File — `loop.go` (2,736 LOC, 84 funcs, 5+ concerns)
2. 🔴 Long Method — `Run()` (545 LOC) and `executeToolCall()` (328 LOC)
3. 🟡 God Struct / Stamp Coupling — `Options` (41 fields) passed by value everywhere
4. 🟡 Data Clump — redundant `(permissionMode, options)` threaded through 16 signatures
5. 🟡 Duplicated Code — ~5 near-identical `ToolResult` builder functions

## Architectural Style Detected

**Layered / modular monolith.** The wider repo splits cleanly into `internal/agent`
(orchestration), `internal/tools` (tool implementations), `internal/sandbox`
(permission/exec policy), `internal/hooks`, `internal/zeroruntime` (provider/message
types), `internal/streamjson`, `internal/lsp`, `internal/verify`, and `internal/redaction`.
`internal/agent` sits at the top of that stack and composes the others.

## Detailed Findings

### Finding #1: God File — `loop.go`

- **Category:** Cohesion / Architecture
- **Severity:** 🔴 Critical
- **Anti-Pattern:** God Object (file-level)
- **Location:** `internal/agent/loop.go` (2,736 lines, 84 functions)
- **Principle Violated:** SRP, Separation of Concerns
- **Description:** One file owns at least five separable concerns:
  1. **Run loop** — `Run`, `finalAnswerAfterMaxTurns`, stream-stall retry, compaction wiring, model-switch, completion gate.
  2. **Tool execution** — `executeToolCall`, `historySafeToolCalls`, `isRetriableToolError`, retriable-error scrubbing.
  3. **Sandbox permission machine** — ~31 functions matching `sandbox`/`Retry` and ~49 matching `permission` (`buildPermissionEvent`, `requestPermission`, `executeRequestPermissions`, `grant*`, `persist*`, `deniedPermissionResult`, …).
  4. **Hooks** — `dispatchBeforeTool`, `dispatchAfterTool`, `blockedByHookResult`.
  5. **Tool advertisement / schema** — `partitionTools`, `ToolVisible`, `ToolAdvertised`, `schemaToRuntimeMap`, `propertyToRuntimeMap`.
- **Evidence:** `grep -c '^func '` → 84 funcs; concern buckets: permission=49, sandbox/retry=31, schema/partition=7, hooks=3, askuser=3.
- **Recommendation:** Split by concern into sibling files in the same package (no API change, purely physical): `permission.go` (permission events, grants, request/deny/cancel results), `sandbox_retry.go` (the `maybeRetry*` / `sandboxDenied*` cluster), `tool_exec.go` (`executeToolCall` and helpers), `tool_advertise.go` (`partitionTools`, `ToolVisible/Advertised`, schema mapping), leaving `loop.go` with just the run loop. Because everything is package-private and covered by tests, this is a move-only refactor.

### Finding #2: Long Method — `Run()` and `executeToolCall()`

- **Category:** Code / Cohesion
- **Severity:** 🔴 Critical
- **Anti-Pattern:** Long Method, Deep Nesting
- **Location:** `loop.go:107` `Run()` (545 LOC); `loop.go:774` `executeToolCall()` (328 LOC); `loop.go:1723` `executeRequestPermissions()` (110 LOC); `loop.go:2438` `partitionTools()` (105 LOC)
- **Principle Violated:** SRP, single level of abstraction per function
- **Description:** `Run()` interleaves turn iteration, per-turn tool partitioning, proactive compaction, stream-stall retry (`maxStreamStallRetries`), compaction-on-error recovery, the tool-call batch loop, self-correct/acceptance gating, mid-run model escalation, and the headless completion gate. Indentation reaches **7 tab levels** with 52 lines nested ≥5 deep, which is a reliable proxy for too many responsibilities in one scope.
- **Evidence:** `awk` indentation probe over lines 107–652: max tab indent = 7; ≥5-tab lines = 52. `messages = append(...)` appears 13× inside the loop.
- **Recommendation:** Extract cohesive phases into named helpers driven by a small per-turn struct: `buildTurnRequest`, `streamWithStallRetry`, `runToolBatch`, `applyModelSwitch`, `checkCompletionGate`. Target < 80 LOC for `Run()`. For `executeToolCall()`, lift the sandbox-retry decision ladder into the `sandbox_retry.go` helpers so the happy path reads top-to-bottom.

### Finding #3: God Struct / Stamp Coupling — `Options`

- **Category:** Coupling / Cohesion
- **Severity:** 🟡 Warning
- **Anti-Pattern:** God Struct, Stamp Coupling
- **Location:** `internal/agent/types.go:177–285` (108 lines, 41 fields)
- **Principle Violated:** ISP, cohesion
- **Description:** `Options` fuses three unrelated groups: (a) static config (`MaxTurns`, `Model`, `ContextWindow`, `Cwd`, …), (b) **13 `On*` callback hooks** (`OnText`, `OnToolCall`, `OnPermission`, `OnUsage`, `OnToolProgress`, `OnContext`, …), and (c) mutable runtime state (`runPermissions *permissionRunState`). The whole struct is then passed **by value** into dozens of functions, so callees receive a large record when most read one or two fields (classic stamp coupling), and the embedded runtime pointer means the "value" isn't really immutable.
- **Evidence:** `types.go` field-line count = 41; struct span = 108 lines; `Options` appears as a parameter across the majority of `loop.go` signatures.
- **Recommendation:** Group the callbacks into an embedded `Callbacks` (or `Observer`) struct and consider a separate `runState` value threaded explicitly rather than smuggled inside `Options`. Where practical, pass narrow interfaces instead of the whole struct. This is a larger change than #1/#2 — stage it after the file split.

### Finding #4: Data Clump — redundant `(permissionMode, options)`

- **Category:** Coupling
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Data Clump / redundant parameter
- **Location:** 16 signatures in `loop.go` (e.g. `executeToolCall:774`, `sandboxDeniedNetworkRetryEvent:1220`, `buildPermissionEvent:2120`)
- **Principle Violated:** DRY, single source of truth
- **Description:** `permissionMode PermissionMode` is passed alongside `options Options` in 16 functions, but `Options` already has a `PermissionMode` field. `Run()` derives a local `permissionMode` (defaulting `""` → `PermissionModeAuto` at `loop.go:122`) and then forwards it separately forever after. That creates two sources of truth for the same value and bloats every signature in the permission path.
- **Evidence:** `grep -c 'permissionMode PermissionMode, options Options'` = 16 (non-test); `options.PermissionMode` is read only once (at `loop.go:122`).
- **Recommendation:** Normalize the default once (`options.PermissionMode = ...` early in `Run`), then drop the standalone `permissionMode` parameter and read `options.PermissionMode` at the leaves. Removes one argument from 16 functions with no behavior change.

### Finding #5: Duplicated Code — `ToolResult` builders

- **Category:** Code
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Duplicated Code
- **Location:** `toolResultFromPrePermissionReject:1373`, `blockedByHookResult:1454`, `requestPermissionsResult:1923`, `deniedPermissionResult:2031`, `canceledPermissionResult:2068`
- **Principle Violated:** DRY
- **Description:** Several functions each assemble a `ToolResult` with the same call-id/name/status/output boilerplate; `deniedPermissionResult` (37 LOC) and `canceledPermissionResult` (32 LOC) are near-identical apart from the reason/action. `loop.go` has 21 `ToolResult{...}` literal constructions.
- **Evidence:** `grep -c 'ToolResult{'` = 21; the two denial/cancel builders share structure line-for-line.
- **Recommendation:** Introduce one constructor, e.g. `newTerminalToolResult(call, status, action, reason, event)`, and have the denied/canceled/hook-blocked variants delegate to it. Collapses ~120 LOC of near-duplicate builders.

### Finding #6: Long Parameter List / Long Return List — sandbox-retry helpers

- **Category:** Code / Design
- **Severity:** 🟡 Warning
- **Anti-Pattern:** Long Parameter List, Long Return List
- **Location:** `maybeRetryUnsandboxedAfterSandboxRestriction:1102`, `maybeRetryWithNetworkAfterSandboxDenial:1159`
- **Principle Violated:** KISS, ISP
- **Description:** These helpers take **9 parameters** and return **7 and 6 values** respectively (`(tools.Result, *ToolResult, bool, PermissionDecisionAction, string, []string, error)`). A 7-tuple return is hard to read at call sites and easy to mis-order.
- **Evidence:** `grep -nE '\) \(([^)]*,){3,}...'` matched both signatures.
- **Recommendation:** Return a single named result struct (e.g. `retryOutcome{ result, toolResult, retried, action, reason, grantedPerms }, error`). Bundle the shared inputs (`call, tool, args, result, options`) into a `retryContext` value.

## Dependency Graph Analysis

Outbound coupling from `internal/agent` (non-test imports): `internal/zeroruntime` (25),
`internal/tools` (19), `internal/sandbox` (10), `internal/hooks` (3), plus
`internal/streamjson`, `internal/lsp`, `internal/verify`, `internal/redaction` (1–2 each).

- **No circular dependencies** observed — the package consumes lower layers and is not
  imported back by them.
- **Coupling hotspot is internal, not cross-package:** the heavy dependence is
  `loop.go` → `internal/sandbox`, concentrated because the entire permission machine
  lives in `loop.go`. Extracting that machine into `permission.go` / `sandbox_retry.go`
  (Finding #1) localizes the sandbox coupling to those files and keeps the run loop thin.

## Refactoring Roadmap

### Immediate Actions (This Sprint) — low risk, test-backed
1. **Split `loop.go` by concern** into `permission.go`, `sandbox_retry.go`, `tool_exec.go`, `tool_advertise.go` (move-only; Finding #1).
2. **Drop the redundant `permissionMode` parameter** in favor of `options.PermissionMode` after normalizing the default once (Finding #4).
3. **Introduce a single terminal-`ToolResult` constructor** and delegate the denied/canceled/hook-blocked builders to it (Finding #5).

### Short-Term (1–3 Months)
1. **Decompose `Run()`** into per-phase helpers (`streamWithStallRetry`, `runToolBatch`, `applyModelSwitch`, `checkCompletionGate`) targeting < 80 LOC (Finding #2).
2. **Shrink `executeToolCall()`** by pushing the sandbox-retry ladder into `sandbox_retry.go`.
3. **Bundle sandbox-retry params/returns** into small structs (Finding #6).

### Long-Term (3–12 Months)
1. **Group `Options` callbacks** into an embedded `Callbacks` struct and extract runtime state (`runPermissions`) out of the config object (Finding #3).
2. **Consider promoting the permission/sandbox machine to its own sub-package** (`internal/agent/permission`) once it is isolated in its own files, giving it a testable public surface and removing the sandbox-coupling concentration from `loop.go` entirely.

## Appendix: Anti-Pattern Reference

- **God Object/File:** one unit (class/file) with too many responsibilities; > 500 LOC or > 20 members is the usual trip-wire. Remedy: extract cohesive groups.
- **Long Method:** > 50 LOC or > 3 nesting levels; mixes abstraction levels. Remedy: extract same-level helpers.
- **God Struct:** a config/parameter object that accretes unrelated fields. Remedy: group by concern, pass narrow interfaces.
- **Stamp Coupling:** passing a large structure where only a few fields are used. Remedy: focused parameters or smaller interfaces (ISP).
- **Data Clump:** the same parameters travel together everywhere. Remedy: value object — or, here, delete the redundant copy.
- **Duplicated Code:** the same knowledge expressed in ≥ 3 places. Remedy: extract a shared constructor/helper.
- **Long Parameter/Return List:** > 4 params or a large return tuple. Remedy: parameter object / named result struct.
